# 通用技术偏好契约（Technical Preferences Contract）

> 本文件属于 Vibe Code Game Studio 框架源代码，不代表任何消费项目的技术选型。
> 框架核心必须保持项目无关、引擎无关、语言无关。消费项目应在自己的数据目录中声明实际命令、平台约束、性能预算和验证步骤。

## 平台与语言

- **框架核心**：不得假设消费项目使用某种游戏引擎、编程语言、构建系统或目录结构。
- **运行环境**：公共命令契约支持 Python 3.10+；平台入口可以包装同一套语义，但不得改变行为。
- **消费项目**：通过版本化项目清单声明命令、环境要求和产物，不将这些信息写回框架核心。

## 命名约定

- 公共协议、Schema 字段和错误码使用稳定、领域中性的名称。
- 文件和标识符遵循其实现语言的既有风格；框架不向消费项目强加代码命名规则。
- 新品牌默认使用 Vibe Code Game Studio。旧 `ccgs` 名称只可作为明确标记的兼容入口，不得成为新增公共接口的默认名称。
- 禁止在框架核心中出现消费项目名称、专属目录、业务实体或特定引擎 API。

## 安全与仓库边界

- 每个会读取或写入消费项目的命令必须显式解析 `framework_root` 与 `project_root`。
- 诊断、预览和规划默认只读；写入必须使用显式写模式，并在可行时提供 dry-run。
- 写入范围采用默认拒绝、声明允许的策略；不得根据某个引擎的目录习惯推断权限。
- 外部服务通过公共 CLI 或版本化协议接入，不得直接解析、修改消费项目的私有文件。
- 日志、事件和报告不得泄露绝对路径、密钥、原始私密提示词或未声明的源码内容。

## 执行与可重复性

- 工作流由消费项目声明；框架只理解步骤、依赖、输入、输出、产物、状态和证据。
- 命令执行必须使用参数数组或等价的安全调用方式，禁止把不可信输入拼接成 Shell 文本。
- 相同输入、配置版本和源状态应产生稳定的计划、标识符与结果结构。
- 重试只适用于明确分类的瞬时失败；业务失败、配置失败和策略失败不得盲目重试。
- 写入必须原子化或具备等价的一致性保证；重复调用必须可安全重放。

## 性能与资源预算

- 框架不内置某类项目的帧率、内存、实体数或渲染预算。
- 超时、并发、日志上限、检索数量和产物保留策略由版本化配置声明，并有安全默认值与边界校验。
- 执行器必须支持取消、超时和有界输出，避免失控任务占用 Worker。

## 测试

- 框架测试只使用人工构造的中性 fixture；不得复制真实消费项目作为模板。
- 必测契约包括：路径安全、Schema 兼容、dry-run 无写入、幂等重试、错误分类、跨平台语义一致和插件隔离。
- 引擎或项目专属验证属于消费项目，不得进入框架核心测试预期。
- 发布前必须运行完整框架测试，并对公共 Schema、CLI 和迁移行为进行兼容性验证。

## 禁止模式

- 在核心代码、默认配置、测试金文件或文档模板中硬编码项目名称、引擎名称、业务目录或项目命令。
- 依据目录名称猜测消费项目类型并自动启用特殊行为。
- 允许外部编排器绕过公共策略、状态机或 Evidence 契约直接修改项目。
- 在 Git 中提交外部服务密钥、机器绝对路径或用户私有数据。
- 将“适配器存在”描述为“外部服务已部署”或“完整闭环已运行”。

## 扩展接口

- 扩展通过版本化 Port/Adapter 契约实现。
- 框架可提供通用适配器，但不得提供以某个游戏引擎或真实项目为默认模板的内置行为。
- 消费项目可以声明自己的命令与策略；这些声明不得改变框架核心的安全边界。

## 架构决策日志

- 框架源 ADR：`.ccgs-core/docs/architecture/`
- 产品目标与需求追踪：`ccgs-data/project-docs/architecture/`
- 所有新增强制规则必须能追踪到 Accepted ADR 或本文件的明确条款。

## 专家路由

- 框架核心不绑定任何游戏引擎专家。
- 专家或工具选择由消费项目配置和当前任务领域决定。
- 未配置路由时使用通用技术、QA、安全和工具链角色，不得从项目目录推断特定引擎。
